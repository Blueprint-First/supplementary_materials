Supplementary reference · positioning

SCA and CodeAct operate at different layers and compose

A worked side-by-side on a single shared task. SCA controls which step runs next with deterministic, expert-authored flow and pre/post-condition guarantees. CodeAct-style generation controls how a single bounded step is executed via LLM-emitted code. The two sit on different layers of the agent stack and can be combined.

Shared task used in all three panels

Natural-language request against a production database: "Mark the order with id O-7741 as cancelled and refund the customer."

  • must not delete a row
  • state transition must be allowed by order_status_fsm
  • refund only if payment_state = "captured"
  • destructive write requires explicit confirmation
  • SQL must type-check against live schema
A · CodeAct alone

LLM emits executable code for the whole task

Action space = code. The model decides both the plan and the code that carries it out.

  1. Model reads the request and current context. LLM
  2. Model emits a single program covering lookup, status update, and refund. LLM
  3. Interpreter runs the emitted code against the live database. EXEC
  4. Model observes the result and decides whether to continue. LLM 
# LLM-emitted action (single turn)
import db
order = db.query("SELECT * FROM orders WHERE id='O-7741'")
# model picks the update shape on its own:
db.execute("UPDATE orders SET status='cancelled' "
           "WHERE id='O-7741'")
db.execute("INSERT INTO refunds(order_id, amount) "
           "VALUES ('O-7741', " + str(order.total) + ")")
print("done")
risk No architectural check that payment_state = "captured". If captured-only is implicit in the prompt, the model may skip it on a sample.
risk "No-delete" / "explicit confirmation" constraints rely on the model following the system prompt; violation is silent until audit.
debug If the run is wrong, failure is diffuse: the trace must be replayed and inspected to localize the offending decision.
B · SCA alone

Deterministic blueprint with pre/post-conditions

Expert-authored program decides the path. Each node has explicit guards. The LLM is invoked only for narrow sub-tasks.

  1. Parse request → structured intent {order_id, action}. LLM
  2. Fetch order; precondition: row exists. DET
  3. Check FSM transition; precondition: (current → cancelled) allowed. DET
  4. Refund branch; guard: payment_state == "captured". DET
  5. Confirm + write; postcondition: status == "cancelled" and audit row exists. DET 
@node(BP)
@precondition(lambda ctx: ctx["order"]["payment_state"] == "captured")
@postcondition(lambda ctx: ctx["refund_id"] is not None)
def issue_refund(ctx):
    # deterministic: parameterized call, no free-form SQL
    ctx["refund_id"] = refunds.create(
        order_id=ctx["order"]["id"],
        amount=ctx["order"]["total"],
    )
guarantee A missing precondition (e.g. payment_state != "captured") halts the engine at the offending node with a localized error.
guarantee "No row deleted" and "explicit confirmation" are architectural, not advisory: the path through the blueprint cannot bypass them.
limit Open-ended sub-tasks (e.g. emitting non-trivial SQL, free-form summarization) still benefit from flexible code generation.
C · SCA + CodeAct

SCA controls flow; one node delegates to CodeAct-style generation under contract

The flow is still deterministic. Inside one bounded node, an LLM emits code under a typed contract enforced by the engine.

  1. Same blueprint as B controls flow and constraints. DET
  2. Inside compose_audit_query, delegate to CodeAct-style code generation. LLM
  3. Engine type-checks the emitted SQL against the live schema (postcondition). ENGINE
  4. If the postcondition fails, the engine retries the node with feedback; after N tries, it aborts locally. ENGINE 
@node(BP)
@precondition(lambda ctx: ctx["schema"] is not None)
@postcondition(lambda ctx: typecheck(ctx["sql"], ctx["schema"])
                and is_read_only(ctx["sql"]))
def compose_audit_query(ctx):
    # Bounded CodeAct-style call: free-form code, fixed contract.
    ctx["sql"] = codeact_subagent.generate(
        task="SELECT audit rows for this order",
        inputs={"order_id": ctx["order"]["id"],
                "schema": ctx["schema"]},
        output_schema={"sql": str},
    )
composition SCA's flow guarantees still hold: the destructive write path is unchanged.
composition CodeAct-style generation is used exactly where it helps — emitting one read-only SQL query under a typed postcondition.
composition A bad generation surfaces locally at this node, not as a downstream side-effect.

Why they compose: different layers of the agent stack

Layer
Who decides
Workflow control
Which node runs next, what guards must hold, when to halt. SCA
Constraint enforcement
Pre / post-conditions are checked by the engine; violations are localized. SCA engine
Sub-task execution
How a single bounded step (e.g. emit SQL, summarize, transform) is performed. CodeAct-style
Tool / interpreter
Runs the emitted code or the deterministic call against the environment. runtime
